## Findings

### F1 — `RENDER_API_KEY` lived in the application's `.env` · **High (operational)** · ✅ resolved

`.env` carried a Render platform API key that the application never read: it appeared nowhere in `src/`, and
`env.schema.ts` did not declare it, so Zod stripped it. A deployment-platform credential with account-wide
reach sat in the file the API loads at boot, for no functional benefit. It never reached the repository
(`.env` is git-ignored and was never committed), but it was printed in plaintext during a session, so it had
to be treated as compromised.

**Resolved** on 2026-07-23: the owner regenerated the key and revoked the old one on the Render dashboard, and
**deleted the `RENDER_API_KEY` line from `.env`** entirely. The key now lives only in the MCP client
configuration, not in the app's environment.

### F2 — No continuous integration · **Medium** · ✅ resolved

There was no `.github/workflows`: everything that guards quality — lint, three test suites, a type-checking
build — had to be remembered and run by hand.

**Resolved** in `.github/workflows/ci.yml`, on push and pull request to `main`. Two independent jobs, so a
lint failure cannot hide an e2e failure: `quality` runs lint, the unit suite and the build; `integration`
brings up Elasticsearch and Redis from the repo's own compose file (`--wait` for healthy), seeds the index and
runs the integration and e2e suites, dumping container logs if anything fails.

Fixing this exposed a second, quieter problem: `npm run lint` carries `--fix`, so in CI it would have repaired
an auto-fixable violation and reported success. Verified — a badly formatted file made `lint` exit **0** after
rewriting the file, while the new no-autofix `lint:ci` exited **1**. CI runs `lint:ci`.

### F3 — No rate limiting on a public read API · **Medium** · ✅ resolved

Every endpoint was a public unauthenticated `GET` fanning out to metered Elasticsearch and Redis, with the
existing guards limiting a request's *shape* (page size ⇒ 400, result window ⇒ 422) but not its *rate*. The
load test showed one laptop driving 1,593 req/s unopposed.

**Resolved** by the `add-request-rate-limiting` OpenSpec change: a global guard (`@nestjs/throttler`) with
per-endpoint budgets (60/min search and suggest, 300/min autocomplete, 120/min default), `/health` exempt.
The counter is in Redis for a shared limit and falls over to an in-process counter on a Redis outage — never
failing open, never failing closed, so Redis stays non-critical (D14). Over-budget returns the typed 429 with
`RateLimit-*` and `Retry-After` headers. Verified under load: with a 60-budget flood, exactly 60 served and
240 refused with a clean typed 429, rejection at 4.4 ms p95 — a flood cannot be turned into a slow-down. The
guard adds no measurable overhead when disabled (`search_cold` p95 30.4 ms vs the 29.7 ms baseline).

### F4 — The coverage number understated what is actually tested · **Low (reporting)** · ✅ resolved

`npm run test:cov` reported **64.76%** statements, which a reader would reasonably read as a third of the code
being untested. The gap was an artifact of the setup: coverage runs only the unit suite, while the files
sitting at 0% were pure wiring exercised only by e2e/integration — `main.ts`, the feature/config modules, the
client factories and lifecycles, and the DTOs.

**Resolved** two ways. `collectCoverageFrom` now excludes the genuinely non-unit-testable wiring (`main.ts`,
`**/*.module.ts`, `**/dto/**`, the client factories and `*-client.lifecycle.ts`, and the seed CLI), so the
figure reflects the logic unit tests target. And the files with real logic that were only e2e-covered now have
unit specs: `app.setup.ts` (CORS resolution and proxy trust), the four thin controllers, and the rate-limit
guard. The result is **97.6% statements across 173 unit tests**, with no logic file left in the dark — an
honest number rather than a caveated one.

### F5 — Direct dependencies sit one major behind · **Informational**

`npm outdated` lists ten majors available: `@elastic/elasticsearch` 8→9, `zod` 3→4, `jest` 29→30, `eslint`
9→10, `typescript` 5.9→7, and the type packages. None is a security issue (`npm audit` is clean), and the
Elasticsearch client **should** stay on 8.x to match the 8.17 server. This is recorded as a deliberate state,
not a defect.
