## Findings

### F1 — `RENDER_API_KEY` lives in the application's `.env` · **High (operational)**

`.env` carries a Render platform API key that the application never reads: it appears nowhere in `src/`, and
`env.schema.ts` does not declare it, so Zod strips it. A deployment-platform credential with account-wide
reach is sitting in the file the API loads at boot, for no functional benefit.

It never reached the repository — `.env` is git-ignored and has never been committed — so the exposure is
local. It was nevertheless **printed in plaintext during this session**, so the key must be treated as
compromised.

**Do:** rotate the key now, and keep it in the MCP client configuration rather than the app's `.env`.

### F2 — No continuous integration · **Medium**

There is no `.github/workflows`. Everything that guards quality — lint, three test suites, a type-checking
build — has to be remembered and run by hand. For a submission judged on engineering practice, a pipeline
running `npm run lint && npm test && npm run build` on every push is cheap and visible; the e2e and
integration suites can run in the same workflow with `docker compose up -d elasticsearch redis` plus a seed
step, exactly as documented.

**Do:** add a workflow before submitting. Not added here — it is beyond the scope of an audit.

### F3 — No rate limiting on a public read API · **Medium**

Neither `@nestjs/throttler` nor any equivalent is present. Every endpoint is a public unauthenticated `GET`
that fans out to Elasticsearch, and `/search` can request up to `SEARCH_MAX_PAGE_SIZE` (100) documents with
aggregations per call. The load test showed a single laptop driving 1,593 req/s at will; against metered
Elastic Cloud and Upstash that is a quota-burn and availability risk, not just a performance one.

The existing guards limit the *shape* of a request (page size ⇒ 400, result window ⇒ 422) but not the *rate*.

**Do:** consider a global throttler, or document the omission as a deliberate scope decision for the challenge.

### F4 — The coverage number understates what is actually tested · **Low (reporting)**

`npm run test:cov` reports **64.76%** statements, and a reader would reasonably read that as a third of the
code being untested. The gap is an artifact of the setup: coverage runs only the unit suite, while the files
sitting at 0% are the composition roots — `main.ts`, `app.setup.ts`, `app.module.ts`, the five feature modules
and `config.module.ts` — which by construction can only be exercised by the e2e suite, and that suite is a
separate Jest project with no coverage collection.

The behaviour is covered; the metric just cannot see it.

**Do:** either exclude the wiring from `collectCoverageFrom` so the figure reflects testable code, or collect
coverage across the e2e project too. Until then, quote the number with its caveat.

### F5 — Direct dependencies sit one major behind · **Informational**

`npm outdated` lists ten majors available: `@elastic/elasticsearch` 8→9, `zod` 3→4, `jest` 29→30, `eslint`
9→10, `typescript` 5.9→7, and the type packages. None is a security issue (`npm audit` is clean), and the
Elasticsearch client **should** stay on 8.x to match the 8.17 server. This is recorded as a deliberate state,
not a defect.

2. **Add a CI workflow** — the highest-visibility improvement left for a reviewer (F2).

4. **Fix the coverage reporting** so the number means what a reader thinks it means (F4).
